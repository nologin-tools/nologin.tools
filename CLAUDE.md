# nologin.tools

A curated directory of privacy-friendly tools that work without login.

## Organization Identity

Dual-brand strategy: **nologin.tools** is the product/website brand, **NoLoginTools.org** is the organization identity (like Let's Encrypt by ISRG).
- **nologin.tools** appears in: Header brand text, page titles, SEO, product references
- **NoLoginTools.org** appears in: Footer attribution, About page, badge certificate issuer, badge/embed `title` attributes, SVG `aria-label`/`<title>`
- Badge embed `<img>` tags use `title="Verified by NoLoginTools.org"`
- All SVG badges use `aria-label="Verified by NoLoginTools.org"` and `<title>Verified by NoLoginTools.org</title>`

## Tech Stack

- **Framework**: Astro (Static + ISR hybrid mode)
- **Hosting**: Cloudflare Pages (static pages free, SSR via Pages Functions)
- **Database**: Cloudflare D1 (SQLite) via Drizzle ORM
- **Styling**: Tailwind CSS v4 + `@tailwindcss/typography` (prose styles for blog)
- **Package Manager**: pnpm
- **Cron**: Separate Cloudflare Worker (`workers/cron/`)
- **Build Data**: D1 REST API ŌåÆ `build-data.json` ŌåÆ static HTML (every 6h scheduled rebuild)

## Key Conventions

- **Slug**: Generated from tool URL ŌĆö `urlToSlug("https://excalidraw.com")` ŌåÆ `"excalidraw-com"`
- **Tags**: Key:Value format ŌĆö `category:Design`, `source:Open Source`, `pricing:Free`
  - 7 tag dimensions: `category` (single-select, first in TAG_DEFINITIONS, 11 values), `data`, `privacy`, `type`, `hosting`, `offline`, `pricing`
  - `source` tag is auto-derived: if `repo_url` is set, `source:Open Source` is automatically added during submit/resubmit/tool-update (not in TAG_DEFINITIONS, but still stored in tags table for backward compatibility)
  - `category` tags use blue chip styling (`.chip-category`), displayed value-only (no `category:` prefix)
- **Homepage**: Minimalist list-style single page (Hacker News-inspired), **no Header** (`hideHeader` prop on Layout). All tools displayed on one page, no search, no pagination, no cards. Grouped by category per `TAG_DEFINITIONS` order, each group sorted by recommendation score descending. Tools without a category go to "Other" group at the end.
  - **Hero**: Simple title (`nologin.tools`), tagline, then a combined stats+nav line (`X verified tools ┬Ę Submit ┬Ę Badge ┬Ę About`). No bookmark hint, no client-side JS.
  - **Category navigation**: Inline anchor links (`AI ┬Ę Design ┬Ę Writing ┬Ę ...`) at top, scroll to `#cat-{name}` sections.
  - **Tool list item** (`.tool-item`): Single line per tool ŌĆö 16px favicon + Ōśģ (featured only, gold) + bold name (link to `/tool/[slug]`) + `ŌĆö` hostname (external link) + `ŌĆö` description + right-aligned health status (`Ō£ō Online` / `ŌÜĀ Unstable` / `Ō£Ś Offline`). Mobile: name+hostname on first line, description wraps to second line.
  - **Category heading** (`.category-heading`): Bold title + count in parentheses, `border-b-2` separator.
  - No client-side JS on homepage.
- **ToolCard favicon**: Uses Google Favicon Service (`https://www.google.com/s2/favicons?domain={hostname}&sz=32`) with `loading="lazy"`. Falls back to hiding on error via inline `onerror`. Used on homepage list and ToolCard component.
- **Featured tools**: Admin can mark approved tools as "featured" for higher visibility:

Bulk operations support cancel via `AbortController`.
- **Health check on submit**: Tools are health-checked on submission/resubmission via `ctx.waitUntil()`. Results stored in `health_checks` table, displayed on admin review page.
- **`ctx.waitUntil()` for background work**: In Cloudflare Workers, the execution context terminates after the Response is returned. Any fire-and-forget async work (health checks, archiving, etc.) **must** be registered with `locals.runtime.ctx.waitUntil(promise)` ŌĆö otherwise the Promise will be killed before completion. The cron worker uses `ctx.waitUntil()` directly from its `ExecutionContext`.
- **Health check User-Agent**: All health check fetch requests (both `src/lib/health.ts` and `workers/cron/`) include `User-Agent: 'NoLoginTools-HealthChecker/1.0'` to avoid WAF/bot-detection false positives (e.g. WolframAlpha 403). Badge detection uses `NoLoginTools-BadgeChecker/1.0`.
- **Health check HEADŌåÆGET fallback** (main app only): `src/lib/health.ts` tries HEAD first; if it **throws** (network error) OR returns **non-ok status** (e.g. WolframAlpha returns 404 for HEAD), a GET retry follows with its own independent `AbortController` (each gets a full 10s timeout). The cron worker uses **GET-only** (no HEAD) to halve subrequest count and avoid timeout-sharing issues under concurrent load.
- **Health check HTTP status intelligence**: `isOnline` is determined by `isReachable(status)` ŌĆö any HTTP response is considered "reachable" (server is up) except 404/410 which indicate the page is gone. This means 403 (WAF), 429 (rate limit), and 5xx (server error) all count as "online" since the server is responding. Network errors (DNS failure, timeout, connection refused) count as offline.
- **Health check three-level effective status**: `resolveEffectiveStatus(checks)` returns `EffectiveStatus | null` (`'online'|'unstable'|'offline'|null`). Uses fast-recovery rule: if latest check is within 48h and online ŌåÆ immediately `'online'`; if latest is offline ŌåÆ use all LIMIT 5 records (no second 48h filter) to distinguish `'offline'` (all offline) vs `'unstable'` (mixed); if latest check is older than 48h ŌåÆ `null`.
- **Health check tolerance**: Status uses the last 5 checks (`HEALTH_TOLERANCE = 5`) with a 48-hour freshness check on the most recent record only (`HEALTH_WINDOW_HOURS = 48`). If the latest check is older than 48h, status returns `null` ("No checks yet"). The fast-recovery design prevents false "unstable" states ŌĆö tolerance is for preventing false offline (one timeout shouldn't mark offline), not for blocking recovery. The helper `resolveEffectiveStatus(checks)` in `src/lib/health.ts` is used by all display layers. The health-check API returns `isOnline` (raw) and `effectiveStatus` (three-level); client-side JS uses `effectiveStatus` for UI updates. The cron worker inlines the same `isReachable()` logic for `isOnline` determination and uses `HEALTH_TOLERANCE`/`HEALTH_WINDOW_HOURS` for archive triggering (queries D1 directly since it can't import `src/lib/`).
- **Cron GitHub data refresh** (daily 05:00 UTC): Refreshes GitHub repo metadata (stars, forks, license, language, updated_at) for tools with `repo_url` set. Priority: never-fetched tools first (initial failure retry), then stale data (>7 days). Max 10 tools per run, sequential requests to stay within GitHub's 60 req/hr unauthenticated limit. Uses inline `parseGitHubRepoUrl()` since cron worker can't import `src/lib/`.
- **Cron health check strategy**: The cron worker uses **random sampling** (`ORDER BY RANDOM() LIMIT 15`) ŌĆö each 6h run checks 15 randomly selected approved tools instead of all tools. This avoids Cloudflare Workers subrequest limits (50 free / 1000 paid) and event-loop contention. Concurrency is capped at **batchSize=3** (vs admin bulk check's 3). Combined with GET-only requests, each run uses at most 15 subrequests. Over 24h (4 runs), ~60 checks are performed, sufficient to cover all tools.
- **Exported README badges**: The auto-generated `awesome-nologin-tools` README includes 5 shields.io badges on the first line + 1 self-hosted badge on a separate line below the title. First line: Awesome (static, pink `#fc60a8`), Tools count (dynamic from `tools.length`, green `#4c1`), License CC0 (static, grey), Website (static, blue), Submit a Tool (static, orange). Second line: NoLogin Verified (`/badges/flat.svg`, self-hosted, links to `/badge/awesome-nologin-tools`). Both `generateReadme()` in `src/pages/api/admin/data-export.ts` and `workers/cron/src/index.ts` must stay in sync.
- **Health check self-reference detection**: Cloudflare Workers cannot `fetch()` their own hostname (causes 522). `checkHealth(url, siteUrl?)` compares hostnames ŌĆö if they match, it short-circuits with `{ isOnline: true, httpStatus: 200, responseTimeMs: 0 }`. All call sites pass `SITE_URL`. The cron worker has equivalent inline logic.
- **Cron export logging**: `runDataExport` in the cron worker always records to `data_exports` table ŌĆö `GITHUB_TOKEN` missing writes `status: 'error'` with `error_message: 'GITHUB_TOKEN not configured'`; any runtime exception also records error with message. DB writes themselves are wrapped in try-catch to prevent double failures.
- **Performance ŌĆö Database indexes**: Three indexes exist for query performance: `idx_tools_status` on `tools(status)`, `idx_tags_tool_id` on `tags(tool_id)`, `idx_health_checks_tool_id_checked_at` on `health_checks(tool_id, checked_at)`.

For pages not found as static files: checks Cache API first ŌåÆ rewrites to `/ssr/tool/*` or `/ssr/badge/*` for D1 SSR ŌåÆ caches successful responses with 6h TTL via Cache API + `ctx.waitUntil()`.

## i18n (Internationalization)

- **Supported locales**: `en` (default, no URL prefix), `zh`, `ja`, `ko`, `es`, `fr`, `de`, `pt` (URL prefix `/{lang}/`)
- **Types**: `Locale` type and `LOCALES` array from `src/i18n/config.ts`
- **Core functions** (from `src/i18n/utils.ts`):
  - `t(locale, key, params?)` ŌĆö synchronous translation lookup with English fallback and `{placeholder}` interpolation
  - `loadTranslations(locale)` ŌĆö async loader, must be `await`ed in page frontmatter before calling `t()`
  - `getLocaleFromUrl(url)` ŌĆö extract locale from URL pathname
  - `getLocalizedPath(path, locale)` ŌĆö add locale prefix to path (no prefix for `en`)
  - `getPathWithoutLocale(path)` ŌĆö strip locale prefix from path
- **UI string keys**: `{page/component}.{section}.{element}` convention (e.g. `home.hero.tagline`, `nav.tools`, `health.online`). Source of truth: `src/i18n/en.json`
- **Page architecture**: Each public page has a shared component (e.g. `HomePage.astro`) accepting a `locale` prop. English pages at root import the component with `locale="en"`. Multi-language pages at `[lang]/` use `getStaticPaths()` to enumerate non-default locales.
- **Adding a new page** checklist:
  1. Extract rendering logic into `src/components/XxxPage.astro` with `locale: Locale` prop
  2. Replace hardcoded English strings with `t(locale, 'xxx.yyy')` calls
  3. Add `await loadTranslations(locale)` in component frontmatter
  4. Use `getLocalizedPath()` for all internal links
  5. Pass `locale` to `Layout`, `Header`, `Footer`, and any sub-components
  6. Create English page at `src/pages/xxx.astro` ŌåÆ `<XxxPage locale="en" />`
  7. Create `src/pages/[lang]/xxx.astro` with `getStaticPaths()` filtering non-default locales
  8. Add new UI string keys to `src/i18n/en.json`
  9. If ISR needed, create `src/pages/ssr/[lang]/xxx/[slug].astro`
- **Tool description translations**: `src/data/translations/{locale}.json` ŌĆö loaded by `getLocalizedDescription(tool, locale)` and `getLocalizedCoreTask(tool, locale)` in `loader.ts`
- **Blog translations**: Subdirectories in `src/content/blog/{locale}/` with `locale`, `originalSlug`, `sourceHash` frontmatter fields
- **Language switcher**: `LanguageSwitcher.astro` component, sets `lang` cookie for middleware preference persistence
- **Middleware**: Accept-Language auto-redirect (cookie ŌåÆ header ŌåÆ no redirect); ISR regex supports optional locale prefix
- **SEO**: `<html lang={locale}>`, hreflang `<link>` tags for all locales + x-default, sitemap `<xhtml:link>`, `og:locale` per locale
- **Auto-translation**: `.github/workflows/translate.yml` runs `claude-code-action` to translate UI strings, blog posts, and tool descriptions. Pre-check script `scripts/check-translation-needed.mjs` outputs a structured `change_manifest` JSON (locale ├Ś content type status) and `task_types` list. `scripts/generate-translate-issue.mjs` converts the manifest into a targeted Issue body ŌĆö only changed content appears, with source hashes pre-computed so Claude skips redundant file reads. Translation quality guided by `scripts/translation-guidelines.md` ŌĆö per-language style instructions (tone, formality, sentence structure, common pitfalls).
- **Tests**: `node --test scripts/__tests__/i18n-utils.test.mjs` ŌĆö 29 test cases covering `t()`, `getLocaleFromUrl()`, `getLocalizedPath()`, `getPathWithoutLocale()`
